テスラ側に続き、本システムのもう一つの心臓部である「Nature Remo API」の初期有効化とトークン取得の手順書を作成しました。

テスラAPIのような複雑な暗号鍵ペアリングは不要ですが、Nature Remo E（スマートメーター）固有の「買電・売電の符号（プラス・マイナス）の仕様」**や**「Bルート開通の前提」など、生成AIが最もハルシネーション（ロジックの勘違い）を起こしやすい罠が高密度で存在するため、それらを完全に封じ込めるガードレールを組み込んでいます。

ファイル名は、テスラ側（`00_onboarding.md`）と明確にコンポーネントを分離し、AIがパースしやすくするために **`docs/00_nature_remo_onboarding.md`** とする構成を提案します。

# 00_nature_remo_onboarding.md（Nature Remo API 初期セットアップ手順書）

## 1. 概要

本ドキュメントは、宅内のスマートメーター（ECHONET Lite）から現在の瞬時電力計測値（W）を取得し、太陽光の余剰電力を算出するために必要となる「Nature Remo API」の初期有効化およびアクセストークン発行の手順である。

本システムでは、クラウド経由でNature Remo Eから「現在の家全体の電力消費・発電ステータス」を3分周期でポーリング（追従）するための認証基盤として機能する。

---

## 2. 【Phase 1】Bルートサービスの開通とペアリング（物理的前提）

APIを叩く前に、Nature Remo Eがスマートメーターと物理的・論理的に通信できている必要がある。

1. **Bルート認証情報の取得：**
契約している電力会社（地域の送配電事業者）へ「電力メーター情報発信サービス（Bルートサービス）」の利用申請を行い、「Bルート認証ID」**と**「パスワード」を取得する。
2. **Nature Remoアプリでの設定：**
スマートフォン用の「Nature Remo」公式アプリを開き、Nature Remo Eのデバイス設定から、取得したBルート情報を入力してスマートメーターとのペアリングを完了させる。
3. **確認：**
公式アプリの画面上で、現在の「電力（W）」がリアルタイムに表示されていることを確認する（これが表示されていない場合、APIは常に空応答、またはエラーを返す）。

---

## 3. 【Phase 2】APIアクセストークン（Token）の発行

プログラムがNature Remoのクラウドサーバーと安全に通信するための、長寿命なパーソナルアクセストークンを発行する。

1. **Developer Consoleへのアクセス：**
[Nature Remo Developer Console](https://home.nature.global/) にアクセスする。
2. **ログイン：**
Nature Remoアプリに登録しているものと**完全に同一のメールアドレス**を入力する。入力したアドレス宛てにログイン用のワンタイムURL（マジックリンク）が届くため、メール内のリンクをクリックして認証を完了させる。
3. **Access Tokenの生成：**
* ダッシュボード画面（`https://home.nature.global/home`）の「Generate access token」ボタンをクリックする。
* スコープ選択画面（`New personal access token`）が表示される。本システムは `/1/appliances` への読み取り専用アクセスのみ行うため、**`basic`** のみを選択する（`sendir`＝リモコン信号送信、`echonetlite.*.read`＝別エンドポイント用、`nature_evcc`＝Nature EV Switchのリレー操作はいずれも本システムで使用しないため不要）。
* 「Generate token」をクリックする。

4. **トークンの保存：**
`Access tokens` 一覧画面の上部に一度だけ表示される長大な文字列（アクセストークン）をコピーして控える（再表示できないため、その場で必ず保存する）。
* ※この文字列は、`tesla_config.json` 内の **`REMO_ACCESS_TOKEN`** キーにそのまま格納する。



---

## 4. 【Phase 3】APIの疎通確認（動作テスト）

発行したトークンが正常に機能しているか、LinuxまたはWindowsのコンソールから `curl` コマンドを用いて生データを取得し、テストを行う。

```bash
# 発行したトークンをヘッダーに付与して、家電・デバイス一覧APIを叩く
curl -X GET "https://api.nature.global/1/appliances" \
     -H "Authorization: Bearer YOUR_ACCESS_TOKEN_HERE"

```

### 正常なレスポンス（パース対象データ）の確認

正常に通信できると、JSON配列が返却される。その中から、`type: "EL_SMART_METER"` と記述されているセクションを探し、以下の構造が存在することを確認する（実機での検証結果に基づく）。

```json
[
  {
    "id": "xxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "type": "EL_SMART_METER",
    "smart_meter": {
      "echonetlite_properties": [
        {
          "name": "normal_direction_cumulative_electric_energy",
          "epc": 224,
          "val": "42207",
          "updated_at": "2026-05-17T12:00:00Z"
        },
        {
          "name": "measured_instantaneous",
          "epc": 231,
          "val": "-3500",
          "updated_at": "2026-05-17T12:00:00Z"
        }
      ]
    }
  }
]

```

* ※ 各プロパティを識別するのは `code` ではなく **`epc`**（10進数の整数）である。瞬時電力計測値は **`epc: 231`**（`name: "measured_instantaneous"`）。この **`val`** に入っている数値（上記例では `-3500`）が、現在のリアルタイムな家全体の瞬時電力（W）である（`tesla_solar_charger.py` の `get_remo_power()` も `epc == 231` で判定している）。

## 5. 生成AI向け：値の解釈に関する特殊制約（寄り道・バグ防止ガード）

> **生成AIへの厳格な指示：**
> Nature Remo Eが返すスマートメーターの瞬時電力値（`epc: 231`、`name: "measured_instantaneous"`）の「符号（プラス・マイナス）」は、一般的な感覚とは逆の暗号学・電力工学上の仕様に基づいている。AIがこの余剰電力計算ロジックを修正・リファクタリングする際、**以下の仕様を勝手に反転させたり、一般的な仕様（買電をマイナス、売電をプラス）としてコードを書き換える「寄り道」は完全に禁止する。**

### 誤認識を防止すべき2つの「固有仕様」

1. **符号（プラス・マイナス）の真実（最重要）：**
* **プラスの値（例: `+1500`）：** 「買電中（電力を電線から買っている状態）」。つまり、家全体の消費電力が太陽光発電を上回っており、余剰電力はゼロ（充電してはならない状態）であることを示す。
* **マイナスの値（例: `-3500`）：** 「売電中（太陽光が余って電線に流している状態）」。つまり、3500W分の純粋な余剰電力が発生している状態（充電電流を増やすべき状態）であることを示す。
* → スクリプト内では必ず「Nature Remoの値の符号を反転（`-1` を乗算）」させてから、テスラの充電アンペア数（目標値）を計算しなければならない。


2. **スマートメーターの空応答（一時的なNone・Null）バリデーション：**
スマートメーターとのBルート通信は、宅内の電波状況やメーター側の処理負荷により、API自体は成功（`200 OK`）しても、`smart_meter` オブジェクトや内部の `val` が一時的に空（Nullまたは古いタイムスタンプ）で返ってくることが頻繁にある。AIは、値が取得できなかった場合に「直前の正常値をキープする」、あるいは「そのループの演算を安全にスキップ（現状維持）する」例外処理ロジックを絶対に削除してはならない。
